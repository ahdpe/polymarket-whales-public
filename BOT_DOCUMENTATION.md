# Полная спецификация бота Polymarket Whales (BOT_DOCUMENTATION)

## 📌 Обзор (Что делает бот)
Polymarket Whales — это многофункциональный бот-аналитик, который в реальном времени мониторит блокчейн (Polygon) и API трейдинговой платформы Polymarket для выявления аномальной активности, крупных сделок ("китов") и паттернов инсайдерской торговли. 

Бот автоматически анализирует сделки, накапливает статистику по кошелькам участников и публикует отформатированные алерты (сигналы) в Telegram и Twitter, а также отображает системную статистику на встроенном веб-дашборде.

---

## 🎭 Роли и сценарии работы

Бот работает непрерывно, разделяя задачи на фоновые процессы. Ключевые паттерны, которые он отслеживает:

1. **🐳 Whale Alerts (Разовые крупные сделки)**
   - Фильтрация одиночных сделок огромного объема. Если объем превышает заданный порог (min_alert_usd).
   - Публикация сигнала о том, что конкретный "Кит" зашел на определенный рынок.

2. **🚨 Insider Alerts (Инсайдерские сигналы)**
   Сложные алгоритмические проверки, которые буферизируют сделки и ищут скоординированные или подозрительные наборы покупок:
   - **CLUSTER (Кластерная активность):** Множество свежих (недавно созданных) кошельков одновременно совершают сделки в одном направлении (исходе) в узком временном окне. 
   - **ACCUMULATION (Долгосрочное накопление):** Набор кошельков медленно, в течение нескольких дней, аккумулируют долю в одном исходе, пока их общий объем не превысит крупный порог (например, $50,000).
   - **BURST (Резкий всплеск):** Внезапный скачок объема покупок на конкретный исход от нескольких независимых кошельков в течение 1-2 часов.

3. **📌 Saved Whales / Saved Markets (Персонализация)**
   - Пользователи могут сохранять (подписываться на) конкретные кошельки или интересующие их рынки, чтобы получать PUSH/уведомления об активности по ним.

4. **🐦 Twitter Queue Service (Постинг в X/Twitter)**
   - Бот поддерживает постинг сигналов в Twitter с помощью специализированной очереди. Учитывает rate limits API, имеет механизм задержки (delay) и повторных попыток.

5. **🌐 Status Dashboard (Веб-интерфейс)**
   - Внутренний Flask-сервер (работающий под WSGI Waitress), раздающий UI-дашборд (на порту 5000), где отображаются: зависшие сигналы, статистика по очередям (queue status), общие объемы и настройки.

## ⚙️ Команды и Основные Скрипты
*Проект имеет четкую структуру, состоящую из основного цикла, воркеров и вспомогательных скриптов.*

- `run.sh [dev|prod|current]` — Базовый скрипт для запуска бота. Менеджерит `.env` файлы, завершает старые процессы `main.py` и поднимает новые через `nohup`.
- `main.py` — Точка входа. Инстанцирует Polymarket API-хендлеры, инициализирует Telegram/Twitter воркеры и запускает асинхронный event loop.
- `services/insider_alerts.py` — **Ядро алгоритмов инсайдерских сигналов**. Содержит всю логику выявления `CLUSTER`, `ACCUMULATION`, `BURST`.
- `services/polymarket.py` — Слой взаимодействия с Data API Polymarket, кэширование, вычисление `Hold` и `Cost` позиций, агрегация сделок (TradeAggregator).
- `storage/alerts_storage.py` (или `alerts_storage.db`) — База данных на SQLite, в которой хранятся все потенциальные инсайдерские паттерны (pending) и уже опубликованные события.
- `scripts/...` — Скрипты для отладки. Например, ручные проверки API, пересыл застрявших твитов, сброс состояний.

## 🎯 Фильтры и алгоритмы выявления

При формировании **Insider** сигналов применяются следующие фильтры (конфигурируются через `user_settings.json`):
- `min_usd` (Минимальный размер отдельной сделки)
- `min_total_usd` (Общая минимальная сумма потраченная всеми кошельками в паттерне)
- `min_wallets` (Минимальное количество уникальных кошельков в сговоре)
- `min_direction_pct` (Порог направленности сговора, например > 70% средств должны идти в один исход, чтобы считать это перекосом).
- `wallet_age_hours` (возраст кошелька. Используется в Cluster для вычисления «свежерегов»).
- `max_positions` (Отсекаем старых юзеров с диверсифицированным портфелем, ищем тех, у кого 1-3 позиции — это значит, что кошелек создали "под конкретную ставку").

### Механика расчета объема позиции (Hold vs Cost)
Для каждого кошелька внутри инсайдерского паттерна бот использует специальный механизм (Position Tracking):
- **Cost (Cost Basis; total_cost):** Реальная цифра в USD, которую кошелек *потратил* на покупки. (Рассчитывается как `vol * p`).
- **Hold (Current Position Value):** Текущая *рыночная стоимость* доли (shares), которую содержит кошелек на момент проверки (`max(trade_sum, current_position_usd)`).
- Бот всегда берет актуальный прайс по позиции из `currentValue` в Polymarket API и перерассчитывает долю `(18%)`.

## ⛔ Исключения и ограничения

1. **Cooldown (Кулдауны):** 
   - На одном рынке по конкретному исходу алерт может сработать максимум 1 раз за заданный период (обычно `24h`). Это защита от спама.
   
2. **Double check API errors (Рейт-лимиты Polymarket):**
   - API Polymarket агрессивно возвращает `429 Too Many Requests`. Слой `polymarket.py` содержит retry-механизмы, семафоры (`_api_semaphore`) и fallback-кэши. Запрос на позицию иногда может упасть, тогда бот берет "raw trades sum" из буфера, чтобы не пропустить алерт.
   
3. **Ghost Positions (Пустые/проданные позиции):**
   - Бот может считывать сделки, но кошелек мог уже все продать. Существует механизм «Drop out» (Confirm wallet has no position), который перепроверяет рынок несколько раз с паузами, прежде чем "сбросить" буфер сделок для кошелька (чтобы предотвратить спам-алерты по проданным активам).

4. **Shared Funding:**
   - Бот может анализировать, финансируются ли различные кошельки инсайдеров из одного Polygon-источника. На такие паттерны вешается флаг `⚠️ Shared funding detected`.

## 🔍 Тонкости, о которых должен знать программист (ИИ)

- **Atomic Published Checks:** Выше было много гонок (race conditions), когда 2 воркера одновременно проверяли базу на наличие уже опубликованного паттерна и пытались отправить сигнал в Телеграм дважды. База SQLite использует `try_mark_published_atomic()` перед отправкой сообщения, это **нельзя ломать!** Сигнал отправляется строго 1 раз.
- **Очереди Twitter / Telegram:** Архитектура имеет раздельные очереди. Telegram-посты пишутся быстро, Twitter — ложится в JSON delay-файлы из-за строгих Twitter-лимитов.
- **Waitress vs Flask dev:** Внутренний веб-сервер должен подниматься только через `Waitress`. dev-сервер Flask блочил потоки и выдавал `Connection reset by peer`. 
- **Кодировки и Символы:** В Telegram используются MarkdownV2-допустимые (или классический MD) спецсимволы. Любое экранирование скобок `()` в ссылках или звездочек `*` для жирного шрифта критично для работы бота (если сломать разметку, Telegram API вернет ошибку `HTTP 400`).
- **Синтетические Trade Объемы:** В `insider_alerts.py` функция `_apply_position_tracking` подменяет "чистые" сделки (trades) на "синтетические", агрегированные по доминантному направлению (исходу) для каждого кошелька. Из-за этого нужно аккуратно протаскивать атрибуты `cost_basis_usd`, `total_shares` и `min/max prices` через синтетический объект, чтобы они не потерялись к моменту рендеринга текста.

## 🚫 ЧЕГО КАТЕГОРИЧЕСКИ НЕЛЬЗЯ ЛОМАТЬ (Red Flags)

1. **Никогда не сбрасывай `total_shares` и расчет `min/max prices`** в `_apply_position_tracking`. Иначе Telegram-презентация превратится в кашу, потеряется процент покупок и PnL.
2. **Не трогай `confirm_wallet_has_no_position`** (особенно параметры rounds). Polymarket API часто врет и выдает пустой ответ; если доверять первой ошибке, бот будет постоянно сбрасывать реальные паттерны, думая, что юзеры "продались".
3. **Осторожнее с `asyncio.Semaphore`**. Проект использует семафоры для ограничения конкурентных вызовов к Data API Polymarket. Если убрать семафоры, Cloudflare моментально забанит IP бота.
4. **Формат строк Telegram:** Имена юзеров `(username)` могут содержать спецсимволы. Их нельзя пушить "как есть" в Markdown без обработки или оборачивания (сейчас они выводятся либо через усечение хеш-адреса, либо обрезаются).
5. **Не меняй `VACUUM` в SQLite-хендлерах.** База сделок (trades window) перезаписывается сотни раз в секунду. Если выключить `PRAGMA wal_checkpoint(TRUNCATE)` и `VACUUM`, жесткий диск заполнится логами за несколько часов.

## ✅ Примеры хороших и плохих кейсов

**🟢 ХОРОШИЙ КЕЙС (Как должен выглядеть Insider сигнал)**
```text
🚨 Potential Insider (CLUSTER)
Market: [Will BTC break 100k?]
Outcome: YES
Window: 2h
Fresh Wallets: 5
Volume: $100,500 (100% YES)
⚠️ Shared funding detected

Participants:
• ⚠️ [0xabc...1234] | 0h | 1 pos
  Hold: $20,280 (18%)
  Cost: $15,000 (12-15%)
```
*Почему хорошо?* Есть корректная ссылка, есть четкое разделение Hold и Cost (даже с интервалом в %, если кит покупал лесенкой), есть бейдж "Shared funding".

**🔴 ПЛОХОЙ КЕЙС (Сломанный код)**
```text
🚨 Potential Insider 
Market: BTC
Outcome: YES
Volume: $20000

Participants:
• Sei6 - $0
```
*В чем ошибка?* Отвалилась связка `currentValue` с API (показывает $0). Сломалась кластеризация `total_shares` (не видно %). Упал кулдаун, и сообщение отправилось 5 раз подряд из-за отсутствия атомарного коммита SQLite. *Никогда не допускать такого поведения.*
